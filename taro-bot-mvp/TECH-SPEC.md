# Техническая спецификация MVP: сатирическое коллекционное Таро в Telegram

Дата: 04.07.2026. Статус: черновик для старта разработки.
Контекст и обоснование идеи — см. `research/tarot-numerology-deep-research-report.docx` и `research/tarot-numerology-market-research-brief.md` в этом репозитории.

## 0. Проверка перед стартом

В Supabase organization `zvekigwnpwbbxbmzwjfa` уже есть проект `leopuzz7-byte's Project` (ref `thjyvcjzurjbufvthlvb`) с таблицами `profiles`, `analyses`, `method_prices`, `orders` — это боевой проект life-cod-v2 (20 профилей, 123 анализа, 39 заказов на момент проверки).

Рекомендация: **не** класть таро-бота в этот же проект. Схема данных другая (Telegram-пользователи, карты, паки, а не веб-профили и анализы), и любая миграция для бота не должна иметь шанс задеть боевые данные основного продукта. Завести **отдельный Supabase-проект в той же организации** — так проще управлять биллингом и держать общие конвенции (RLS включён везде, как в текущем проекте), но данные и миграции изолированы. Могу создать его через Supabase MCP, когда подтвердишь.

## 1. Архитектура

- **Бот:** TypeScript + grammY (современный, активно поддерживаемый фреймворк для Telegram Bot API; лучше python-telegram-bot вписывается в существующий TS-стек life-cod-v2 — переиспользуются тайпинги, конвенции, часть кода).
- **Хостинг бота:** Vercel Serverless Function на webhook (`POST /api/telegram/webhook`) — переиспользует уже настроенный Vercel-проект и деплой-пайплайн (projectId `prj_mzIOx2jm531koNV6253cuGFaF5N1` из CLAUDE.md), не нужен отдельный always-on процесс.
- **БД:** Supabase Postgres (новый проект, см. п.0), RLS включён на всех таблицах.
- **AI:** GPT-4o-mini через proxyapi.ru — та же связка, что уже используется в life-cod-v2 (`useAIReading`/`usePersonalReading` паттерн), тот же аккаунт/прокси.
- **Рендер картинки для шеринга:** `@napi-rs/canvas` (Node, без headless-браузера) — рендерит карту + текст + рамку в PNG прямо в serverless-функции, без Puppeteer (тяжело для serverless).
- **Платежи:** Telegram Bot Payments API с `currency: "XTR"` (Stars) — нативно, не нужен внешний эквайринг для MVP.

## 2. Схема данных (Postgres/Supabase)

```sql
create table public.tarot_users (
  id uuid primary key default gen_random_uuid(),
  telegram_id bigint unique not null,
  username text,
  vibe_preference text default 'sarcastic', -- sarcastic | unhinged_friend | deadpan
  is_pro boolean default false,
  pro_until timestamptz,
  streak_count int default 0,
  last_draw_at timestamptz,
  daily_chat_count int default 0,
  daily_chat_reset_at timestamptz default now(),
  created_at timestamptz default now()
);

create table public.tarot_cards (
  id serial primary key,
  name text not null,
  arcana text not null,          -- major | minor
  base_meaning text,
  rarity text default 'common',  -- common | rare | epic | legendary | meme_exclusive
  image_url text
);

create table public.tarot_card_variants (
  id serial primary key,
  card_id int references public.tarot_cards(id),
  variant_name text,             -- альтернативный "мемный" скин карты
  rarity text,
  image_url text
);

create table public.tarot_user_cards (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references public.tarot_users(id),
  card_id int references public.tarot_cards(id),
  variant_id int references public.tarot_card_variants(id),
  obtained_at timestamptz default now(),
  source text                    -- daily | pack | event | duel
);

create table public.tarot_daily_draws (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references public.tarot_users(id),
  card_id int references public.tarot_cards(id),
  reading_text text,
  draw_date date default current_date,
  shared boolean default false,  -- ключевая метрика валидации, см. п.7
  shared_at timestamptz
);

create table public.tarot_chat_messages (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references public.tarot_users(id),
  role text,                     -- user | assistant
  content text,
  created_at timestamptz default now()
);

create table public.tarot_purchases (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references public.tarot_users(id),
  product_type text,             -- pack | battle_pass | chat_cap_removal
  stars_amount int,
  telegram_payment_charge_id text,
  created_at timestamptz default now()
);

create table public.tarot_packs (
  id serial primary key,
  name text,
  price_stars int,
  guaranteed_min_rarity text
);
```

Хранить `chat_messages` с TTL/автоочисткой (например, последние 20 сообщений на пользователя) — учитывая репутационные кейсы утечек в отчёте (Sanctuary, Moonly), не копить лишние личные данные без необходимости.

## 3. Флоу бота (MVP v1)

1. `/start` → онбординг в 2-3 сообщениях, выбор `vibe_preference` (кнопки: "дерзко", "по-дружески токсично", "с холодным сарказмом") → создаётся `tarot_users`.
2. **Карта дня** (главная кнопка в меню) → генерация расклада (см. п.5) → рендер PNG → отправка с кнопкой "Поделиться в Stories" (deep link на `share_daily_card`, помечает `shared=true`).
3. **Коллекция** → инлайн-список открытых карт (`X/78 + N мемных вариантов`), прогресс-бар.
4. **Чат с тарологом** → свободный диалог с персонажем (system prompt из п.4), лимит 5 сообщений/день на free, дальше платно.
5. **Магазин** → паки карт за Stars, battle pass (см. п.6).

v1.5 (после теста v1): **Дуэль** — два пользователя тянут по карте, GPT одной репликой в шутку решает "победителя" с абсурдным обоснованием; результат тоже шерabelen.

## 4. GPT-персонаж: черновик system prompt

```
Ты - самоироничный ИИ-таролог в Telegram-боте. Ты не веришь всерьёз в
предсказания и не скрываешь этого, но подаёшь расклады с абсолютной
уверенностью и характером - как подруга, которая "гадает на картах"
просто потому что это смешно и в кассу.

ТОН: дерзкий, ироничный, в меру токсичный по-доброму, в теме актуальных
мемов и сленга. Короткие реплики - это Telegram-чат, а не эссе (2-4
предложения на ответ). Никогда не звучи как корпоративный ассистент.

ГРАНИЦЫ (обязательны, не обсуждаются):
- Если пользователь прямо спрашивает "это по-настоящему?" - честно
  говори, что это развлекательный формат, без притворства настоящим
  предсказателем, психологом или врачом.
- Не давай всерьёз медицинских, юридических, финансовых советов - можно
  пошутить на тему, но при реальном запросе мягко направь к специалисту.
- Шутки про стереотипы (внешность, свидания, работа) - в дружелюбном
  ключе, без реальной жестокости и без дискриминации по защищённым
  признакам (национальность, религия, инвалидность и т.п.) - это не
  смешно, а токсично, и должно быть под жёстким запретом.
- Если пользователь пишет о суицидальных мыслях, самоповреждении, абьюзе
  или похожих темах - НЕМЕДЛЕННО выходи из образа, отвечай серьёзно и
  по-человечески, предложи обратиться за реальной помощью. Не шути в
  этот момент вообще.
```

Это черновик, а не финальный текст — перед запуском рекомендую прогнать через живого человека с чувством юмора (см. п.8, риск "AI-юмор выдыхается") и добавить 10-15 few-shot примеров реплик в нужном тоне прямо в промпт — это стабилизирует голос сильнее, чем инструкции.

## 5. Генератор "карты дня" - структура (не полностью открытая генерация)

Чтобы шутки не были вымученными и повторяющимися, слоты фиксированы, GPT заполняет только переменные части:

```
Шаблон: {персонаж/тип_человека} + {место/сценарий} + {модификатор-панчлайн}

Карта: [название карты, например "Башня"]
Базовое значение карты: [взято из tarot_cards.base_meaning]

GPT получает: карту + базовое значение + 3-5 last_draws пользователя
(чтобы не повторяться) → возвращает 1-2 предложения в формате
"сегодня ты [сценарий]" в заданном tone of voice.

Пример вывода (карта "Башня", vibe = sarcastic):
"Сегодня ты встретишь альтушку на остановке, и это будет самое
стабильное событие твоего дня - дальше по расписанию Башня, все
планы рушатся, зато будет что рассказать".
```

Хранить пул использованных сценариев/панчлайнов последних N дней на пользователя в промпте (anti-repeat), а не полагаться на "случайность" модели.

## 6. Монетизация - конкретика на MVP

| Продукт | Цена (Stars, ориентир) | Что даёт |
|---|---|---|
| Пак из 5 карт (мемные скины) | ~150-200 Stars | Косметические варианты карт, без игрового преимущества |
| Battle Pass (месяц) | ~250-350 Stars/мес | Эксклюзивные мемные карты сезона + безлимит чата + 2 карты дня вместо 1 |
| Снятие лимита чата (месяц) | ~100-150 Stars/мес | Безлимитные сообщения тарологу |

Курс Stars к валюте меняется — перепроверить актуальный курс перед финальным прайсингом. Free tier: 1 расклад/день, 5 сообщений чата/день, коллекция открывается только через ежедневные расклады (без паков) - паки чисто ускоряют/дают редкие скины, не единственный путь.

Реализация приёма: `bot.api.sendInvoice(..., currency: "XTR", prices: [{label, amount}])`, обработка `successful_payment` → запись в `tarot_purchases` → апдейт `tarot_users`/выдача `tarot_user_cards`.

## 7. Метрика валидации MVP (проверить до вложений в игру/паки)

Главный вопрос v1: **пересылают ли люди карту дня без напоминаний**. Инструментировать:
- `tarot_daily_draws.shared` + `shared_at` - клик по кнопке "Поделиться".
- Целевой ориентир для решения "строить дальше": ≥15-20% DAU шерят карту хотя бы раз в неделю. Ниже - переделывать контент/голос до вложений в паки и дуэли, не масштабировать раньше времени.
- Второй по важности сигнал - `streak_count` (retention): держат ли пользователи серию открытий 7+ дней подряд.

## 8. Дорожная карта

- **Недели 1-2:** каркас бота, `/start`, карта дня + рендер PNG + шеринг. Без GPT-чата, без паков. Цель - проверить п.7.
- **Недели 3-4:** GPT-чат с персонажем (лимит 5/день), базовая коллекция (карта дня = автоматически пополняет коллекцию, без гачи).
- **Недели 5-6:** Stars-платежи, паки, battle pass.
- **v1.5+ (после ретеншн-данных):** async-дуэль между пользователями.
- **v2 (не раньше подтверждённого PMF):** полноценная Mini App с игровым слоем (по образцу Hamster Kombat), см. предыдущее обсуждение.

## 9. Технические и содержательные риски

- **Стоимость AI под контролем** - GPT-4o-mini на 6-7К токенов/генерацию - доли цента, но лимиты на free tier всё равно нужны против абьюза (спам-запросы).
- **Модерация контента** - даже сатира не должна скатываться в реально оскорбительное; минимум - словарь стоп-тем/стоп-групп в system prompt (см. п.4) плюс выборочный ревью логов на старте вручную.
- **Голос персонажа - не чисто инженерная задача** - без живого редактора с чувством юмора, курирующего промпт и few-shot примеры, есть риск получить "корпоративный юмор", который никто не будет шерить (см. предыдущее обсуждение в чате).
- **Приватность чатов** - шифрование/минимизация `tarot_chat_messages`, TTL на хранение, не логировать лишнее.
- **Разделение брендов** - если life-cod-v2 держит серьёзное позиционирование по нумерологии, у бота должно быть отдельное имя/персонаж, не суб-фича существующего бренда (риск из предыдущего обсуждения).
